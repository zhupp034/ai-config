# Global Conventions

- 当用户定义了术语解释时，在总结、报告和生成文档中优先采用这些解释。
- 术语映射应尽量维护在可持续更新的 glossary 文件中，避免直接硬编码到 prompt 或生成文本里。
- 优先使用共享术语表 `/Users/zpp/.codex/memories/term-glossary.json`；skill 本地 glossary 仅作为兜底或 skill 专用扩展使用。
- 如果项目内存在相应的 Markdown 说明文件（如 `README.md`、模块说明、使用文档等），每次改动后都应评估是否需要同步更新；如未更新，需说明原因。

# Git Commit Rules

- Git 提交信息必须使用 Conventional Commits。
- 允许的 type: feat, fix, docs, refactor, test, chore, build, ci.
- 提交信息格式：`type(scope): summary`；如果没有明确 scope，可使用 `type: summary`。
- `summary` 使用英文，动词短语，小写开头，尽量不超过 72 个字符。
- 未经用户明确要求，不主动执行 `git commit`。
- 如需提交，先检查变更范围，避免把无关文件一并提交。
- 对文件内容变动的总结不能遗漏关键改动，至少覆盖实际变更的主要文件和行为影响。
- 汇报提交内容、变更摘要、文件变动总结时，默认使用中文输出，除非用户明确要求其他语言。

# Frontend Rules

- 前端代码优先统一使用 TypeScript。
- 禁止新增裸 `any`；确实需要使用时，必须在代码中写明原因，并将影响范围控制在最小。
- 前端代码生成时，优先复用项目内已有组件、基础样式体系和封装好的接口请求方式，避免重复造轮子。
- 颜色、间距、圆角、层级统一走 design token 或 CSS 变量。
- 样式实现优先复用项目现有的 CSS 主题色 token、设计变量或统一样式变量，不新增与既有主题体系冲突的颜色常量。
- 禁止编写高耦合选择器，避免随意覆盖全局样式。
- 涉及用户操作链路的改动，必须覆盖 loading、empty、error、success 等关键状态。
- 新页面和核心组件默认检查桌面端与移动端表现。
- 有设计稿时优先按设计稿实现，避免擅自偏离既定视觉与交互方案。
- 交付或提交变更时，必须写清用户可见变化、影响范围、验证方式和风险点。
- 涉及 UI 的可见改动，默认提供截图或清晰的前后对比说明。
- 接口数据展示必须处理空值、异常值和重复提交等基础异常场景。
- 新增功能或功能改动时，应同步增加或更新对应的单元测试文件；如未更新，需明确说明原因。
- css样式要最低兼容chrome浏览器
